# ADR 012: Client — Static asset build and CSP-safe config injection

**SPEC:** [client](../specs/client.md)
**Status:** Accepted
**Date:** 2026-03-24

---

## Context

The browser client is currently an inline `<script type="module">` block rendered server-side by `src/server/client.ts`. Config values (`cols`, `rows`, `theme`, `copyOnSelect`, etc.) and the `sessionId` are injected as template literals directly into the script source at request time.

This approach has several problems as the client grows:

1. **No TypeScript in browser code** — the `<script>` block is an untyped string. No LSP, no type checking, no refactoring support.
2. **Inline scripts block CSP** — `Content-Security-Policy: script-src 'self'` is the standard secure default. Inline scripts require `'unsafe-inline'` or a per-request nonce, making strict CSP impossible without extra infrastructure.
3. **Untestable browser logic** — the JS is embedded in a template string. Unit tests can only assert on the rendered HTML, not on the behavior of the code.
4. **No path to enhancement** — adding a settings panel, keyboard shortcuts, or multi-session UI requires growing an inline string. A real TypeScript file with imports is necessary.

## Decision

Split the client into three static assets served directly from `dist/`:

```
dist/
  client.html          ← static HTML shell, served at GET /s/:id
  client-browser.js    ← compiled from src/client/index.ts
  ghostty-web.js       ← existing (ghostty-web package)
  ghostty-vt.wasm      ← existing (ghostty-web package)
```

### Config injection — `GET /api/config`

The compiled `client-browser.js` has no server-side values baked in. On load, it fetches config from a new `GET /api/config` endpoint and reads `sessionId` from `window.location.pathname`.

```
GET /api/config → { cols, rows, fontSize, fontFamily, cursorBlink,
                    scrollback, theme, copyOnSelect, rightClickBehavior }
```

This is the only approach that achieves zero inline script — no `<script>` block with dynamic content, no placeholder substitution, no `__config` global. The HTML is fully static and identical for every session.

**Why a fetch instead of a `<meta>` tag or `<script type="application/json">`?**

A `<script type="application/json">` tag is not executable but is still server-rendered — the HTML would differ per request (or at minimum per config change). A fetch keeps the HTML 100% static, cacheable, and free of any server-side substitution.

**Round-trip cost**: on localhost, `GET /api/config` takes <1ms. The config is fetched once on page load. This is acceptable.

### HTML shell — `src/client/index.html`

A minimal static file:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>webtty</title>
    <link rel="stylesheet" href="/dist/client.css" />
  </head>
  <body>
    <div id="terminal"></div>
    <script type="module" src="/dist/client-browser.js"></script>
  </body>
</html>
```

No inline script. No dynamic content. The `<title>` is updated by `client-browser.ts` after the `sessionId` is known (`document.title = sessionId + ' | webtty'`).

### Browser TypeScript — `src/client/index.ts`

All current browser JS migrated to real TypeScript:

```ts
import { init, Terminal, FitAddon } from '/dist/ghostty-web.js';

interface ClientConfig { ... }

const sessionId = window.location.pathname.split('/s/')[1];
const config: ClientConfig = await fetch('/api/config').then(r => r.json());

document.title = `${sessionId} | webtty`;
// ... rest of terminal init, WebSocket, copy behavior
```

Compiled by `Bun.build()` with `target: 'browser'` into `dist/client-browser.js`.

### CSS — `src/client/index.css`

The inline `<style>` block extracted to `src/client/index.css`, copied to `dist/client.css` at build time. The background color is no longer injected from `theme.background` — the terminal container background defaults to `#000` and `ghostty-web` paints its own background on the canvas.

### Build pipeline — `scripts/build.ts`

Two additional steps after the existing server build:

1. `Bun.build({ entrypoints: ['src/client/index.ts'], target: 'browser', outdir: 'dist' })` — compiles browser TS
2. Copy `src/client/index.html` and `src/client/index.css` to `dist/`

### Routing — `src/server/routes.ts`

`GET /s/:id` changes from calling `render(sessionId, config)` to serving `dist/client.html` as a static file.

`GET /api/config` is a new endpoint — returns `loadConfig()` as JSON, omitting server-only keys (`port`, `host`, `shell`, `term`, `colorTerm`, `logs`).

### `src/server/client.ts`

Removed. The server no longer renders HTML.

## Considered Options

**Option A: Inline script + nonce (keep server-side rendering)**

Generate a fresh nonce per request, inject it into both the `Content-Security-Policy` header and the `<script nonce="...">` tag.

- **Pros**: Keeps config injection synchronous, no extra round-trip, CSP-safe.
- **Cons**: Server must set CSP headers on every response. Nonce must be cryptographically random per request. More complex server-side rendering infrastructure. The client is still a string template — no TypeScript.

**Option B: `<script type="application/json">` for config, external JS**

Inject config as a non-executable JSON tag:
```html
<script type="application/json" id="webtty-config">{"cols":80,...}</script>
```
Client reads `document.getElementById('webtty-config').textContent`.

- **Pros**: No fetch round-trip. CSP-safe (non-executable type). Pattern used by Next.js (`__NEXT_DATA__`).
- **Cons**: HTML is still server-rendered (config changes per request). Not truly static — cannot be cached at a CDN edge. Requires server to still run `loadConfig()` and render HTML.

**Option C: `globalThis.render(config)` — compiled JS + inline call**

Compile `client-browser.ts` to a string, inline it into `<script>`, append `render(config)` as an inline call.

- **Pros**: No extra fetch. Inline config call is small.
- **Cons**: Inline script — still requires `'unsafe-inline'`. The compiled JS is inlined, not served from a URL, so it cannot be cached by the browser separately from the HTML.

**Option D: Chosen — fully static HTML + `GET /api/config`**

- **Pros**: Zero inline script. `script-src 'self'` is sufficient. HTML is identical for every session — cacheable. Client is real TypeScript with LSP, type checking, imports. Clean foundation for future UI work (settings panel, session switcher).
- **Cons**: One extra round-trip (`GET /api/config`) on page load. Acceptable on localhost (<1ms); negligible on LAN.

## Consequences

- `script-src 'self'` is sufficient — no `'unsafe-inline'`, no nonce infrastructure needed.
- Browser JS is real TypeScript: LSP, type checking, imports, unit-testable behavior.
- `src/server/client.ts` is deleted — server is no longer in the HTML rendering business.
- `GET /api/config` is a new public endpoint — returns only client-relevant config keys. Server-only keys (`port`, `host`, `shell`, `term`, `colorTerm`, `logs`) are not exposed.
- HTML is identical for every session — the tab title is set by JS after load (minor: title shows "webtty" for ~1 render frame before JS updates it).
- Background color is no longer injected from `theme.background` into CSS — the canvas paints its own background. The `<html>`/`<body>` background defaults to `#000`, which matches all dark themes and is invisible under the full-viewport canvas.
- The build step is slightly longer (one extra `Bun.build()` call + two file copies).
