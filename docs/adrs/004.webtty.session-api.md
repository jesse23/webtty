# ADR 004: Session API â€” REST endpoints for session CRUD and rename

**SPEC:** [webtty](../specs/webtty.md)
**Status:** Accepted
**Date:** 2026-03-22

---

## Context

webtty needs a way to create, list, inspect, rename, and delete terminal sessions over HTTP. The CLI (`webtty run`, `webtty ls`, `webtty rename`, `webtty rm`) and the browser UI both consume this API. The design must be cross-platform, debuggable with `curl`, and consistent with REST conventions.

## Decision

Add a `/api/sessions` resource to `src/server.ts` with five endpoints:

| Method | Path | Status codes |
|--------|------|-------------|
| `GET` | `/api/sessions` | `200` |
| `POST` | `/api/sessions` | `201`, `400`, `409` |
| `GET` | `/api/sessions/:id` | `200`, `404` |
| `PATCH` | `/api/sessions/:id` | `200`, `400`, `404`, `409` |
| `DELETE` | `/api/sessions/:id` | `204`, `404` |

**ID validation** is enforced on `POST` and `PATCH`: `a-z0-9-_.`, 1â€“64 chars. Invalid IDs return `400`.

**`POST /api/sessions`** accepts an optional body `{ id?: string }`. If `id` is omitted, the server auto-generates an 8-character lowercase hex ID. Returns `409` if the ID already exists.

**`PATCH /api/sessions/:id`** accepts body `{ id: string }` (the new ID). The session's PTY and WebSocket state are preserved â€” only the key in the in-memory registry changes. Returns `409` if the new ID is already taken. The old URL (`/s/:old-id`) becomes a `404` after rename; no redirect.

**`DELETE /api/sessions/:id`** kills the PTY, closes the connected WebSocket (if any), and removes the session from the registry. Returns `204` on success, `404` if absent.

**`GET /api/sessions`** is also the server liveness signal â€” the CLI uses connection refused vs `200` to determine if the server is running, replacing a separate `/health` endpoint.

## Considered Options

### Option A: REST resource (chosen)

Standard HTTP verbs on `/api/sessions`. Consistent with existing `POST /api/server/stop` pattern. Debuggable with `curl`. Works from CLI and browser.

### Option B: WebSocket RPC

Send session management commands over WebSocket. Rejected â€” adds statefulness to session management that doesn't need it. HTTP is sufficient and simpler.

### Option C: `PUT /api/sessions/:id` for rename

`PUT` implies full replacement of the resource. `PATCH` is more precise â€” only the `id` field changes, PTY state is preserved.

### Option D: Separate `/health` endpoint for server detection

Already decided against in [ADR 002](002.cli.start-stop.md). `GET /api/sessions` returning `200` vs connection refused is sufficient.

## Consequences

- CLI commands `webtty ls`, `webtty run`, `webtty rename`, `webtty rm` all have a stable HTTP contract to build against
- Rename does not redirect old URLs â€” open browser tabs on the old session URL will 404; acceptable for a local dev tool
- Session state is memory-only; no persistence across server restarts (deferred to a future slice)
